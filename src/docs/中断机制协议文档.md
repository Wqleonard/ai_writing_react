# InkMind 中断机制前后端协议文档

> 适用范围：`todo`（工具审核）与 `form`（澄清表单）两类中断。  
> 本文档基于当前代码实现，统一说明 `__interrupt__` payload、前端渲染、`resume` 回传格式与后端消费逻辑。

---

## 1. 总体流程

1. Agent 在执行过程中触发中断（`interrupt(...)`）。
2. 流式接口返回 `updates` 事件，chunk 中包含 `__interrupt__`。
3. 前端根据 `__interrupt__` 里的 `type` 渲染对应面板：
   - `type: "todo"` -> 工具审核面板
   - `type: "form"` -> 澄清表单面板
4. 用户在面板里完成操作，前端调用 `resume` 接口恢复运行。
5. 后端读取 `resume` 的 `decisions`，继续后续节点执行。

---

## 2. `__interrupt__` 事件结构

LangGraph 常见外层结构（简化）：

```json
{
  "__interrupt__": [
    {
      "id": "xxx",
      "value": {
        "type": "todo",
        "value": {
          "...": "..."
        }
      }
    }
  ]
}
```

说明：

- `__interrupt__` 是数组，理论上可能出现多个中断项。
- 每一项通常有：
  - `id`: 本次中断标识（前端可用于关联）
  - `value`: 中断 payload
- 我们约定 payload 统一为二层结构：
  - 顶层：`{ type, value }`
  - 业务数据放在 `value` 中

---

## 3. 两类中断 payload 规范

### 3.1 todo（工具审核）

后端发出中断时 payload：

```json
{
  "type": "todo",
  "value": {
    "action_requests": [
      {
        "name": "tool_name",
        "args": {}
      }
    ],
    "review_configs": [
      {
        "action_name": "tool_name",
        "allowed_decisions": ["approve", "edit", "reject"]
      }
    ]
  }
}
```

用途：

- 前端展示工具调用参数
- 用户对每个 tool call 给出决策（批准/编辑/拒绝）

### 3.2 form（澄清表单）

后端发出中断时 payload：

```json
{
  "type": "form",
  "value": {
    "id": "optional-form-id",
    "title": "请补充创作偏好",
    "intro": "请先完成以下选择",
    "questions": [
      {
        "id": 1,
        "type": "single",
        "question": "请选择风格",
        "options": ["轻松", "严肃"]
      },
      {
        "id": 2,
        "type": "mutiple",
        "question": "请选择元素",
        "options": ["海边", "落日", "雨夜"]
      }
    ]
  }
}
```

说明：

- `questions[].type` 当前实现使用 `"single"` / `"mutiple"`（拼写保持兼容现状）。
- `options` 可直接使用字符串数组，前端按字符串值作为提交 answer。

## 4. resume 统一协议（重点）

当前统一约定：**所有 resume 都用 `type + decisions`**。

接口请求（示例）：

```json
{
  "assistant_id": "inkmind",
  "command": {
    "resume": {
      "type": "todo",
      "decisions": [
        { "type": "approve" }
      ]
    }
  },
  "stream_mode": ["updates", "messages"],
  "stream_subgraphs": true,
  "config": {},
  "on_disconnect": "cancel"
}
```

### 4.1 todo 的 decisions

```json
{
  "type": "todo",
  "decisions": [
    { "type": "approve" },
    { "type": "edit", "edited_action": { "name": "tool_name", "args": {} } },
    { "type": "reject", "message": "原因（可选）" }
  ]
}
```

约束：

- `decisions.length` 必须与待审核 `action_requests.length` 一致，否则后端报错。

### 4.2 form 的 decisions

```json
{
  "type": "form",
  "decisions": [
    { "id": 1, "question": "请选择风格", "answer": ["轻松"] },
    { "id": 2, "question": "请选择元素", "answer": ["海边", "落日"] }
  ]
}
```

说明：

- `answer` 单选通常为字符串，多选通常为数组。
- `id`、`question` 会被后端保留，便于后续系统注入与日志追踪。

---

## 5. 后端消费与状态落库

### 5.1 todo

- 中间件读取 `resume.decisions`，逐条映射回 tool call。
- 根据决策更新 AIMessage 中 tool calls，并在拒绝时补充 ToolMessage。

### 5.2 form

- 中间件读取 `resume.decisions`（数组）并写入：
  - `state["clarification_answers"] = decisions`
- 然后清空 `state["pending_form"]`，并 `jump_to: "model"` 继续执行。

---

## 6. 前端渲染约定

中断面板按 `interrupt.type` 分发：

- `type === "form"`：渲染澄清表单面板
- 其他（含 `type === "todo"`）：渲染工具审核面板

当前为了调试可视化：

- `todo` 类型面板使用绿色样式（标题显示 `TODO 审核（type=todo）`），便于确认类型是否生效。

---

## 7. 兼容性与注意事项

1. 新协议已统一到 `decisions`，不要再使用旧的 `answers`。
2. `resume.type` 当前主要用于前端和日志语义标注；后端关键字段是 `decisions`。
3. 若前端做重连或断线恢复，建议保留中断项 `id` 以便关联同一中断。
4. `form` 问题类型 `"mutiple"` 为现存协议值，若后续修正为 `"multiple"`，需前后端同步迁移。

---

## 8. 最小联调清单

1. 触发 `todo` 中断：确认面板为绿色、标题含 `type=todo`。
2. 提交 `todo` 决策：确认 run 从 `interrupted` 恢复继续执行。
3. 触发 `form` 中断：确认题目与选项正常展示。
4. 提交 `form` 决策：确认 `clarification_answers` 可在后续上下文中看到（系统注入文本）。
